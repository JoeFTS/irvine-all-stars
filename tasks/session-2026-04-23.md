## Lessons Learned

- **Per-row pending state beats global busy** — when an optimistic UI has multiple rows with the same action, gate per row. A global `busy` flag turns rapid clicks on different rows into serialization or perceived race conditions.
- **Mark side effects as committed only after dependent work succeeds** — `complete-signup` was marking the invite used unconditionally even when the team_coaches upsert failed, stranding the user. The comment claimed otherwise. Re-read the comments under code review.
- **Trust boundaries belong at the API, not just the UI** — `send-invite` accepted `team_id` without validating it matched `division`. The admin UI filtered correctly but a malicious or buggy client could submit anything. Always validate at the API.
- **Renaming applied migrations is dangerous** — even when naming is inconsistent. Carry forward the convention in the next batch instead.
- **`JSON.stringify` compare for hook identity stability is cheap-but-belt-and-suspenders** — the simpler fix is structuring effects so they don't depend on derived array identity at all.
- **Spawning N parallel fix subagents only works when their file scopes don't overlap** — bundle multiple findings touching the same file into one agent (admin roster page got 4 findings consolidated into one subagent here).
- **Re-run E2E after policy/trigger rewrites** — semantically-equivalent SQL rewrites can still break if the planner makes a different choice. Cheap insurance.
